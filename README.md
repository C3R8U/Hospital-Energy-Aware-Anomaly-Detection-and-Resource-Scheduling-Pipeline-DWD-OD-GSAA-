# Hospital-Energy-Aware-Anomaly-Detection-and-Resource-Scheduling-Pipeline-DWD-OD-GSAA-
This repository provides a fully runnable reference pipeline that mirrors the paper’s end-to-end workflow: the first stage performs anomaly detection using a Dynamic Weighted Distance-Based Outlier Detection method, and the second stage optimizes resource scheduling after isolating anomalous samples using a Genetic Simulated Annealing Algorithm within an MRCOF-style scheduling simulation. The script reads the dataset file `hospital_communication_energy_system.csv` and produces three output artifacts in the chosen output directory: an anomaly scoring table `anomaly_scores.csv`, a scheduling plan `scheduling_plan.csv`, and a run metadata/summary file `summary.json`. The implementation is designed to be reproducible in a standalone Python environment while preserving the key computational structure described in the manuscript.

After loading the data, `prepare_dataframe` performs minimal but essential preprocessing so the downstream algorithms can operate on structured numerical features. The `Timestamp` column is parsed into a standard datetime type. The string-formatted blood pressure field `Blood Pressure (mmHg)` is converted into two numeric columns, `BP Systolic (mmHg)` and `BP Diastolic (mmHg)`, so it can be included in distance-based modeling. Columns such as `Energy Saving Mode`, `System Health Check`, and `Day of the Week` are coerced into numeric types when present. Feature selection is handled by `select_features`, which prioritizes energy consumption, environmental conditions, device power usage, and key vital sign indicators that match the paper’s context; if some preferred columns are missing, it falls back to using all available numeric columns (excluding a small set of non-feature-like fields) to keep the script robust across dataset variants.

The anomaly detection stage implements DWD-OD (Dynamic Weighted Distance-Based Outlier Detection). The central idea is to embed two kinds of information into the distance metric: domain priority of each dimension and the short-term fluctuation intensity of each dimension within a local time window. The code first applies z-score standardization to all selected features to prevent scale dominance across heterogeneous units. It then processes the data using a sliding window where `window_size` controls how many rows are used for local statistics and `step_size` controls how far the window advances each iteration. Within each window, a fluctuation coefficient `beta` is computed as `std/mean` per feature dimension, which reflects relative variability during that window. A priority vector `alpha` is built by `build_alpha_weights`, giving higher weight to clinically and operationally sensitive dimensions such as medical equipment power usage, total power usage, oxygen level, heart rate, and blood pressure. The final dynamic weights `w` combine `alpha` and `beta`, so dimensions that are both important and currently volatile contribute more strongly to the distance calculation.

Distances are computed using an Adaptive Mahalanobis Distance (AMD). The window covariance matrix is estimated and ridge-regularized for numerical stability, then the dynamic weights are injected through a diagonal weight matrix inside the quadratic form. Intuitively, this amplifies differences along “important and fluctuating” dimensions, making anomalous points more separable from normal patterns. For each sample in the window, the algorithm computes the mean AMD distance to its `k_neighbors` nearest neighbors. Larger mean neighbor distance indicates stronger local isolation. The script converts mean neighbor distance into a density proxy `density = 1/(avg_k_dist + eps)`, then forms a LOF-like anomaly score by combining the sample’s average distance with the window’s overall density level. Instead of using a fixed global threshold, a dynamic threshold is computed per window using a quantile rule controlled by `threshold_quantile` (default 0.95). The output includes `anomaly_score`, `threshold`, and a binary `is_anomaly` label for each record, enabling direct inspection of how the decision boundary evolved over time.

The scheduling stage is a runnable engineering representation of the MRCOF (Multimodal Resource Collaborative Optimization Framework) concept and the GSAA (Genetic Simulated Annealing Algorithm) optimization loop. Since real OpenStack/Kubernetes scheduling requires external cluster infrastructure, the code models “deployment” as a task-to-node assignment optimization problem that remains faithful to the intended objective: balancing latency-related cost against energy-related cost. Nodes are created by `build_nodes`, each with an abstract compute capacity, a renewable energy ratio, and a baseline latency. The renewable ratio is sampled around the dataset distribution of `Renewable Energy Usage (%)` so that simulated nodes remain realistic. Node quality is summarized by `energy_efficiency_score`, which combines compute capacity and renewable ratio into a single efficiency measure.

The objective function used by `assignment_cost` is an engineering form of the paper’s weighted multi-objective cost: total cost equals `omega_1 * total_delay + omega_2 * total_energy`, where `w1_latency` and `w2_energy` correspond to `omega_1` and `omega_2`. Energy cost is derived from each task’s `Total Power Usage (kWh)` (or `Energy Consumption (kWh)` if needed), adjusted by node efficiency and renewable ratio so that greener and more capable nodes reduce effective energy cost. Delay cost is modeled as baseline node latency plus a queueing delay term driven by accumulated node load and compute capacity. Task priority is computed by `task_priority_weight` using `System Health Check` and `AI Predicted Health Status`, increasing weight when a record indicates system risk or unhealthy status, and adding an extra delay penalty if such critical tasks are mapped onto low-efficiency nodes. This models the operational preference that critical workloads should avoid unstable or slow resources.

GSAA performs global search using a genetic population while using simulated annealing’s Metropolis acceptance rule to avoid premature convergence. Each individual is an integer vector assigning each task to a node. In each generation, the algorithm uses tournament selection, then applies one-point crossover with probability `pc` and random-reset mutation with probability `pm`. The probabilities are adaptive: `pc` decays exponentially over generations while `pm` gradually increases, encouraging broader recombination early and stronger perturbation later. The simulated annealing temperature follows a geometric cooling schedule from `sa_temp0` down to `sa_temp_min`; when a new candidate is worse, it can still be accepted with probability `exp(-delta/T)`, enabling controlled uphill moves early and more conservative refinement later. The final output is the best assignment vector and its objective value, saved into `scheduling_plan.csv` along with node metadata such as renewable ratio and baseline latency.

The script can be run with defaults or tuned via command-line parameters. The `--csv` argument points to the dataset path, `--out` sets the output directory, and `--seed` fixes randomness for reproducibility. DWD-OD behavior is controlled by `--window-size`, `--step-size`, `--k`, and `--th-quantile`. Larger windows smooth local statistics but may reduce sensitivity to short bursts; smaller steps increase temporal resolution but increase computation; larger `k` stabilizes neighborhoods but can blunt very local anomalies; higher threshold quantiles are more conservative and flag fewer anomalies. Scheduling parameters include `--nodes` for the number of simulated nodes and `--max-tasks` to cap the number of normal tasks scheduled for runtime control. GSAA can be tuned with `--pop` and `--gen` for search strength and `--w1`/`--w2` to adjust the latency–energy trade-off. For paper-style result reproduction, `anomaly_scores.csv` supports anomaly rate and temporal analysis, `scheduling_plan.csv` supports node load and cost analysis, and `summary.json` captures the complete configuration and key statistics for experiment tracking.

Several parts of the implementation are intentionally engineered to match a hospital energy-and-communication context while remaining editable for strict formula alignment. The most direct tuning points are `build_alpha_weights` (domain priority mapping), `task_priority_weight` (criticality modeling), and `assignment_cost` (objective shaping). The DWD-OD structure—dynamic weights, AMD distance, sliding-window scoring, and quantile-based thresholding—along with the GSAA loop—population search plus annealing acceptance—remain stable and preserve the method’s core logic while allowing practical adaptation to different operational environments.
